# DocInsight MVP PRD

## 1. 제품 개요

DocInsight는 사용자가 업로드한 문서를 기준으로 질문을 입력하면, 문서 안에서 관련 근거를 찾고 그 근거를 바탕으로 답변을 생성하는 문서 기반 Q&A 서비스다.

현재 MVP는 한 번의 질문마다 사용자가 업로드한 특정 문서 1개를 대상으로 검색과 답변 생성을 수행한다. 여러 문서가 DB에 저장될 수는 있지만, 질문 시점의 검색 대상은 현재 Q&A 페이지의 `document_id`에 연결된 문서 하나다.

## 2. 사용자

- PDF, TXT, MD 문서를 빠르게 이해하고 싶은 개인 사용자
- 기술 문서, 기획 문서, 운영 문서를 기준으로 빠르게 질의응답하고 싶은 개발자 또는 팀원
- 문서 전체를 직접 읽기 전에 핵심 내용을 먼저 확인하고 싶은 사용자

## 3. MVP 목표

- 사용자가 문서를 업로드할 수 있다.
- 업로드된 문서에서 텍스트를 추출할 수 있다.
- 추출된 텍스트를 검색 가능한 청크로 나누고 임베딩으로 저장할 수 있다.
- 사용자가 특정 문서에 대해 질문할 수 있다.
- 질문과 관련된 문서 청크를 검색할 수 있다.
- 검색된 청크를 근거로 LLM 답변을 생성할 수 있다.
- 답변과 함께 근거 청크를 보여줄 수 있다.
- 검색 기준을 넘지 못한 경우에도 가장 가까운 참고 청크를 fallback 근거로 구분해 보여줄 수 있다.

## 4. 사용자 여정

1. 사용자는 메인 페이지에 진입한다.
2. 사용자는 문서 업로드 페이지로 이동한다.
3. 사용자는 PDF, TXT, MD 파일을 선택해 업로드한다.
4. 시스템은 문서를 저장하고 즉시 처리한다.
5. 시스템은 텍스트 추출, 청킹, 임베딩 생성, 청크 저장을 수행한다.
6. 처리가 완료되면 사용자는 해당 문서의 Q&A 페이지로 이동한다.
7. 사용자는 질문을 입력한다.
8. 시스템은 질문을 저장하고 질문 임베딩을 생성한다.
9. 시스템은 해당 문서의 청크 중 관련도가 높은 청크를 검색한다.
10. 시스템은 검색된 청크를 근거로 답변을 생성한다.
11. 사용자는 답변, 근거, 유사도, fallback 여부를 확인한다.
12. 사용자는 같은 문서에 대해 추가 질문하거나 메인 페이지로 돌아가 새 문서를 업로드한다.

| F006 | 검색 | 질문 임베딩과 문서 청크를 비교해 관련 청크 검색 | 완료 |

## 6. 페이지별 요구사항

### 메인 페이지 `/`

| 항목 | 내용 |
|------|------|
| 역할 | 서비스 진입점 |
| 주요 기능 | 서비스 소개, 문서 업로드 시작 버튼 |
| 다음 이동 | `/upload` |

### 업로드 페이지 `/upload`

| 항목 | 내용 |
|------|------|
| 역할 | 사용자가 분석할 문서를 선택하고 업로드 |
| 주요 기능 | 파일 선택, 선택 파일명 표시, 업로드 실행, 지원 형식 안내 |
| 다음 이동 | 업로드 성공 후 `/processing/{document_id}` |

### 처리 페이지 `/processing/[id]`

| 항목 | 내용 |
|------|------|
| 역할 | 문서 처리 상태 안내 |
| 주요 기능 | 처리 상태 조회, 완료 시 Q&A 페이지 이동, 실패 시 에러 표시 |
| 다음 이동 | 처리 완료 후 `/qa/{document_id}` |

### Q&A 페이지 `/qa/[id]`

| 항목 | 내용 |
|------|------|
| 역할 | 특정 문서에 대한 질문과 답변 수행 |
| 주요 기능 | 질문 입력, 검색, 답변 생성, 근거 표시, fallback 근거 구분, 메인 이동 버튼 |
| 검색 대상 | 현재 URL의 `document_id`에 해당하는 문서 청크 |
| 다음 이동 | 같은 페이지에서 추가 질문 또는 메인 페이지 이동 |

## 7. 검색 정책

현재 검색은 벡터 유사도와 키워드 일치도를 함께 사용하는 hybrid search 방식이다.

| 항목 | 현재 정책 |
|------|----------|
| 기본 검색 대상 | 현재 문서의 모든 청크 |
| 기본 결과 수 | 최대 5개 |
| 최대 limit | 10개 |
| 기본 유사도 threshold | `SIMILARITY_THRESHOLD`, 기본값 `0.5` |
| 키워드 가중치 | `0.2` |
| fallback 조건 | threshold 이상 결과가 0개일 때 |
| fallback 결과 수 | 최대 2개 |

fallback 결과는 답변 생성을 시도할 수 있는 참고 청크이지만, 관련성이 낮을 수 있으므로 UI에서 일반 근거와 구분한다.

## 8. 청킹 정책

현재 기본 청킹 정책은 다음과 같다.

| 항목 | 값 |
|------|----|
| 최소 토큰 | 150 |
| 최대 토큰 | 400 |
| 오버랩 토큰 | 50 |

짧은 문서는 기본 최소 토큰 기준 때문에 청크가 0개가 될 수 있으므로, 이 경우 `minTokens: 1`로 완화해 최소한의 청크를 생성한다.

섹션 제목이 감지되는 문서는 청크 앞에 현재 섹션 제목을 보강해 검색 문맥을 개선한다.

## 9. 핵심 데이터 개체

### Document

- 사용자가 업로드한 원본 문서
- 파일명, 파일 형식, 파일 경로, 처리 상태, 추출 텍스트, 생성/수정 시간을 가진다.

### DocumentChunk

- 문서를 나눈 검색 단위
- 청크 텍스트, 순서, 임베딩, 소속 문서를 가진다.

### Question

- 사용자가 입력한 질문
- 질문 텍스트, 질문 임베딩, 대상 문서를 가진다.

### Answer

- LLM이 생성한 답변
- 질문, 답변 텍스트, 근거 청크 목록을 가진다.

## 10. MVP 제외 범위

- 사용자 인증
- 사용자별 문서 권한 관리
- 여러 문서를 동시에 대상으로 하는 통합 검색
- 멀티턴 대화
- 문서 목록/삭제/관리 UI
- 고급 reranker 모델
- 운영용 rate limit
- 프로덕션 배포 및 모니터링

## 11. 현재 한계

- 답변 품질은 검색된 청크 품질에 크게 의존한다.
- 짧거나 구조가 약한 문서는 유사도가 낮게 나올 수 있다.
- fallback 결과는 항상 정답 근거를 보장하지 않는다.
- 기존에 처리된 문서는 청킹 정책 변경이 자동 반영되지 않으며 재업로드 또는 재처리가 필요하다.
- 현재 저장된 여러 문서는 DB에 남지만, Q&A는 특정 문서 하나를 대상으로 동작한다.

## 12. 성공 기준

- 사용자가 PDF/TXT/MD 파일을 업로드할 수 있다.
- 업로드 후 문서가 `completed` 상태가 된다.
- 문서 청크와 임베딩이 저장된다.
- 사용자가 질문을 입력하면 검색 결과가 반환된다.
- 검색 결과 기반 답변이 생성된다.
- 답변과 근거가 UI에 표시된다.
- fallback 상황에서 일반 근거와 다른 UI로 표시된다.
